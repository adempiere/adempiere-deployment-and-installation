## The control node — orchestration entry point

The **control node** is your own workstation or laptop — the machine from which you run all Ansible commands. It is not a server being deployed. Nothing runs on it permanently once the deployment is complete.

What makes it the central actor:
- All `ansible-playbook` commands and both entry-point scripts (`deploy-backend.sh`, `restore-db.sh`) are executed here
- Ansible connects **outbound** to the servers over SSH and executes tasks remotely — the servers never initiate contact or pull configuration
- All operator-specific files live here: `group_vars/all/vars.yml`, `group_vars/all/vault.yml`, the SSH keypair (`ssh_keys/`), the vault password file (`~/.vault_pass.txt`), and the deployment logs (`logs/`)

What the control node requires:
- Ansible (core 2.14+) and the three required collections (`community.docker`, `community.postgresql`, `community.crypto`)
- SSH access to the servers (the project generates its own keypair)
- No Docker, no ADempiere, no other application software

```
Control node (your laptop/workstation)
│
│  Files:
│  ├── group_vars/all/vars.yml      ← plain-text config (gitignored)
│  ├── group_vars/all/vault.yml     ← AES-256 encrypted secrets (gitignored)
│  ├── inventories/hosts.yml        ← server IPs (gitignored)
│  ├── ssh_keys/                    ← project SSH keypair (gitignored)
│  ├── ~/.vault_pass.txt            ← vault password (never in the repo)
│  └── logs/                        ← deployment logs written here
│
│  Commands:
│  ├── ./deploy-backend.sh          ← full BackEnd provisioning
│  ├── ./restore-db.sh              ← database restore
│  └── ansible-playbook *.yml       ← individual playbook runs
│
└── SSH outbound ──────────────────► BackEnd VPS  (deploys ADempiere)
                                   ► FrontEnd VPS (deploys Traefik)
```

## BackEnd internals — the ADempiere container stack

The BackEnd VPS runs the [adempiere-ui-gateway](https://github.com/Systemhaus-Westfalia/adempiere-ui-gateway) Docker Compose stack. All containers are managed by Docker Compose and share a single internal network.

### What gets deployed

The `deploy-adempiere` role:  
1. Creates the installation directory (`install_path`, default `/opt/development`)  
2. Clones or updates the `adempiere-ui-gateway` repository at the configured branch  
3. Renders the `override.env` file from `roles/deploy-adempiere/templates/override.env.j2` — this injects runtime values (PostgreSQL password, paths) into the Docker Compose environment  
4. Starts the stack with `start-all.sh` (first run initialises the database and pulls all images), waits for health checks, and does a clean restart  

### Container stack (representative subset)

| Container | Role |
|---|---|
| `postgresql` | PostgreSQL database — the ADempiere data store |
| `zk` | ADempiere ZK web application server |
| `vue-ui` | Vue.js frontend UI |
| `nginx-ui-gateway` | nginx reverse proxy inside the stack — routes between UI components |
| `kafka` + `zookeeper` | Message queue for POS and async processing |
| `keycloak-service` | Identity and access management |
| `opensearch` + `opensearch-dashboards` | Log aggregation and search |
| `report-engine` | Report generation service |
| `s3-storage` + `s3-gateway-rs` | Object storage for documents and attachments |

### Data persistence

PostgreSQL data is stored in a named Docker volume. The volume survives container restarts and re-deployments. A `./restore-db.sh` run drops and recreates the database, so the volume contents are replaced.

### Where data lives on disk

```
/opt/development/adempiere-ui-gateway/
├── docker-compose/
│   ├── override.env                     ← generated by Ansible; contains DB password
│   └── postgresql/
│       └── postgres_backups/            ← backup archives uploaded here by restore-db.sh
│           ├── 01-Backupscripts/        ← cron scripts deployed by deploy-crontab
│           └── 02-Logs/                 ← cron stdout/stderr logs
```

### Automatic start and stop

`deploy-crontab.yml` installs three crontab entries under the admin user:
- `@reboot` — start the stack when the server boots
- `23:50` daily — stop the stack (pre-backup window)
- `23:55` daily — restart the stack (after backup)

## FrontEnd internals — Traefik

> ⚠ The Traefik workflow is partially implemented. See [traefik-status.md](traefik-status.md) before enabling.

The FrontEnd VPS runs two containers: `traefik` and `socket-proxy`.

### How Traefik receives routing configuration

Traefik does not read its routing rules from the Docker Compose labels of the ADempiere containers (those run on a different server). Instead, it reads static configuration files from the directory `/docker/traefik/config/` on the FrontEnd server.

When `deploy-traefik.yml` runs, Ansible renders `roles/deploy-traefik/templates/app-adempiere.yaml.j2` and copies the result to `/docker/traefik/config/app-adempiere.yaml`. Traefik watches that directory and applies any file placed there immediately — no restart needed.

### How TLS certificates are issued

Traefik uses the **ACME DNS-01 challenge** to request certificates from Let's Encrypt:

1. Traefik requests a certificate from Let's Encrypt for `<dns_domain>`
2. Let's Encrypt asks Traefik to prove domain ownership by placing a DNS TXT record
3. Traefik calls the Cloudflare API (using `cloudflare_token` from the vault) to create the TXT record
4. Let's Encrypt verifies the record and issues the certificate
5. Traefik stores the certificate at `/docker/traefik/certs/cloudflare-acme.json` and renews it automatically before expiry

This approach does not require port 80 to be reachable during certificate issuance — useful if only port 443 is open.

### The socket proxy

Traefik needs to query the Docker daemon to discover containers. Rather than mounting the Docker socket directly (which would give Traefik full daemon control), `socket-proxy` sits between them and exposes only the read-only API calls Traefik needs. This limits the blast radius if Traefik is ever compromised.

```
Traefik ──────► socket-proxy ──────► Docker daemon
               (read-only API)
```

## End-user perspective — accessing ADempiere

Once both servers are provisioned:

1. **User opens a browser** and navigates to `https://adempiere.<dns_domain>`
2. **DNS** resolves the domain to the FrontEnd VPS IP
3. **Traefik** on the FrontEnd terminates TLS (presents the Let's Encrypt certificate), reads the `Host` header, and matches it against the routing rule in `app-adempiere.yaml`
4. **Traefik forwards** the request over plain HTTP to the BackEnd VPS IP (internal network, not exposed to the internet directly)
5. **nginx-ui-gateway** inside the ADempiere stack routes the request to the correct component
6. **ADempiere** processes the request and returns the response

The user sees a standard HTTPS ADempiere login page. The TLS certificate is from Let's Encrypt and is trusted by all modern browsers.

## Operator perspective — day-to-day operations

### Check that the stack is running

```bash
# SSH to the BackEnd server
ssh <your-admin-username>@<backend_ip> -p <custom_sshport>

# List running containers
sudo docker ps --format "table {{.Names}}\t{{.Status}}"

# Check PostgreSQL specifically
sudo docker inspect --format='{{.State.Health.Status}}' adempiere-ui-gateway.postgresql
```

### Restart the stack manually

```bash
cd /opt/development/adempiere-ui-gateway/docker-compose
sudo ./stop-all.sh
sudo ./start-all.sh
```

### Update ADempiere to a new version

```bash
# Pull the new branch/tag and restart the containers
ansible-playbook deploy-adempiere.yml -e "repo_version=main"
```

### Restore a database backup

```bash
# Set restore_backup_filename and restore_local_dir in group_vars/all/vars.yml, then:
./restore-db.sh
```

See [docs/running.md](running.md) for the full command reference and [docs/operations.md](operations.md) for day-2 tasks.
